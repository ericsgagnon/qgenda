    Introduction

    Accessing the API

    1. The base URL for all QGenda REST API resources is https://api.qgenda.com/v2
    2. Access to the API is only available via HTTPS (SSL)
    3. Authentication method is a token-based approach

    Authentication

    Authentication is required for all requests to the QGenda REST API, unless otherwise noted. Authentication with the REST API is performed by supplying a QGenda username/email via a POST to the login endpoint. This returns a token that must be included in all future requests.

    Authentication failures on subsequent requests will always return an HTTP 401 (Unauthorized) status code.

    Note: The “Content-Type” in the headers for the POST must be “application/x-www-form-urlencoded”

    Response from Authentication:

    The response from the server will contain an access token (“access_token” field), a token type (“token_type” field), and an expiration time (“expires_in” field) represented in seconds. When the token expires, the API user will need to POST again to the login URL and receive a new token.

    All future requests must have an “Authentication” header with the token type and token appended as the value. Example:

    Authorization: bearer eyJhbGciOiJBMjU2S1ciLCJlbmMiO...

    Data Format

    All data returned from the QGenda REST API will be in JSON format, unless otherwise noted. JSON libraries are available for most modern programming languages, and many have support built-in to the core libraries.

    While not required, it is strongly recommended that requests include the HTTP Content-Type header, set to a value of application/json (ex. Content-Type: application/json). If no Content-Type header is specified, then application/json will be the default in most cases. However, setting the Content-Type header to a value other than** appliction/json **is generally not supported by the QGenda REST API.

    The following is a list of some popular JSON libraries for various platforms:

    1. Json.NET (.NET)
    2. JSON Processing/JSR 253 RI (Java)
    3. Google Gson (Java)
    4. JSON for Ruby (Ruby)
    5. JSON for PHP (PHP)

    Compression

    The QGenda REST API supports GZip compression of returned data. This will greatly reduce the size of the data returned by the API, resulting in faster response times and reducing bandwidth usage. The vast majority of modern web browsers and HTTP client libraries support GZip compression. We require that all users take advantage of GZip compression.

    To enable GZip compression, the Accept-Encoding *header must be included in the request, and its value must include either *gzip or **. Multi-values are supported, as long as *gzip is one of the values (i.e. Accept-Encoding: gzip, deflate)

    Example 1 (gzip):

        Accept-Encoding: gzip

    Example 2 (*):

        Accept-Encoding: *

    OData Support

    The Open Data Protocol (OData) provides the ability to query a RESTful service, allowing for data to be retrieved in a multitude of ways from a single resource.

    Some resources within the QGenda REST API support OData parameters, allowing users additional control over the data that's returned from the resource. In cases where OData parameters are supported, the resource documentation will provide additional information.

    NOTE: By definition, all OData parameters, operators, and expressions are case-sensitive. Since OData expressions may include the names of specific object properties, those properties are also case sensitive (for instance, a property named Date would not be the same as a property named date).

    NOTE: For readability purposes, the OData $filter parameter is often shown in examples with spaces and other characters that would need to be properly encoded for actual use in a URL. Users must always make sure that URLs for their requests are properly encoded.

    Schedule

    GET Schedule
    https://api.qgenda.com/v2/schedule?companyKey=GUID&startDate=Date&endDate=Date&includeDeletes=Boolean&sinceModifiedTimestamp=Timestamp&includes=String&$select=OData Expression&$filter=OData Expression&$orderby=OData Expression&$expand=OData Expression

    Returns schedule entries for a specific company and date range.

    Response Object: Array of ScheduleEntry

    Note: If GZip compression is supported/accepted by the client, up to 100 days of schedule data may be requested. If compression is not supported, then up to 31 days of schedule data may be returned.

    Note: Times are displayed according to the task’s location time zone or, if there isn’t a task location time zone, it uses the company’s time zone instead.

    If the schedule entry’s start or end time falls within a time range affected by Daylight Saving Time (DST), the following rules apply:

    • If the start time falls within the “gap” at the start of DST, StartTime will be an hour earlier to account for the hour “lost” by the time change.
    • If the end time falls within the “gap” at the start of DST, EndTime will be an hour later that the schedule entry end time to account for the hour “lost” by the time change.
    • If the start time falls within the “fall back” hour at the end of DST, StartDateUtc and EndDateUtc will use the non-DST UTC.
    • If the end time falls within the “fall back” hour at the end of DST, StartDateUtc and EndDateUtc will use the DST UTC.

    For tracking schedule changes:

    We recommend that you first seed your database for the necessary date range you want to track, by calling this endpoint without sinceModifiedTimestamp. Once that's done, you can take the MAX of the returned LastModifiedDateUTC property of that dataset for that same date range, and provide that as the sinceModifiedTimestamp parameter when you next call GET /schedule to get only changes. Then, continue to take the new MAX of LastModifiedDateUTC for the next call with sinceModifiedTimestamp.

    By default we do not include deleted assignments (aka schedule entries). In order to include deleted assignments, you can send includeDeletes=true and we will return anything deleted for the date range (and since sinceModifiedTimestamp if provided) as entries with their isStruck property = true.

    HEADERS
    Authorization
    bearer eyJhbGciOiJBMjU2S1ciLCJlbmMiO...
    PARAMS
    companyKey
    GUID

    The key for the company associated with the schedule entries

    startDate
    Date

    REQUIRED

    The earliest date to retrieve schedule entries for.

    Values should be in M/D/YYYY or MM/DD/YYYY format (ex. 11/1/2014, 5/23/2013, 01/06/2014)

    Note: If startDate is specified without specifying endDate, then schedule entries will be retrieved for a single day. It is functionally equal to setting the start and end date to the same value

    endDate
    Date

    The latest date to retrieve schedule entries for. All dates between startDate and endDate will be included.

    Values should be in M/D/YYYY or MM/DD/YYYY format (ex. 11/1/2014, 5/23/2013, 01/06/2014)

    Note: If either endDate is not provided or if the number of days between the start and end dates is greater than the 100 days, then the endDate will be set so that the number of days from start to end is equal to 100 days.

    includeDeletes
    Boolean

    If provided, deleted schedule entries will be included. IsStruck will be set to true.

    sinceModifiedTimestamp
    Timestamp

    If provided, only schedule entries that have been changed after sinceModifiedTimestamp. Date format is YYYY-MM-DDThh:mm:ssZ UTC ISO-8601. (Example: 2019-05-07T18:50:51.9239331Z)

    includes
    String

    If provided, as a comma separated list of entities the specified related entities will be returned with the response. (Examples: StaffTags,TaskTags,LocationTags)

    GET Rotations
    https://api.qgenda.com/v2/schedule/rotations?companyKey=GUID&rangeStartDate=Date&rangeEndDate=Date&ignoreHoliday=Boolean&ignoreWeekend=Boolean&$select=OData Expression&$filter=OData Expression&$orderby=OData Expression&$expand=OData Expression

    Returns schedule rotations for a specific company and date range. Rotations will be followed to their beginning/end even if the dates for those boundaries are outside the rangeStartDate and rangeEndDate range provided (up to 6 months before rangeStartDate and 6 months after requestEndDate).

    Resource URL: https://api.qgenda.com/v2/schedule/rotations

    Returns: An array of ScheduleRotation objects, in JSON format

    PARAMS
    companyKey
    GUID

    The key for the company associated with the schedule entries

    rangeStartDate
    Date

    The earliest date to retrieve schedule entries for.

    Values should be in M/D/YYYY or MM/DD/YYYY format (ex. 11/1/2014, 5/23/2013, 01/06/2014)

    rangeEndDate
    Date

    The latest date to retrieve schedule entries for. All dates between startDate and endDate will be included.

    Values should be in M/D/YYYY or MM/DD/YYYY format (ex. 11/1/2014, 5/23/2013, 01/06/2014)

    Note: If the number of days between the start and end dates is greater than the maximum number of days allowed (380), then the endDate will be changed so that the number of days from start to end is equal to the maximum number of days.

    ignoreHoliday
    Boolean

    When ignoreHoliday is set to true, the calculation of rotations will ignore days that would break the rotation that are QGenda holidays

    ignoreWeekend
    Boolean

    When ignoreWeekend is set to true, the calculation of rotations will ignore days that would break the rotation that are weekend days. Note: Weekends are defined for the QGenda Company as Fri, Sat or Sat, Sun.

    Future Changes and API Versioning

    QGenda enhances our REST API as new features are introduced in our product. All API changes are categorized into either non-breaking changes or breaking changes. QGenda reserves the right to make non-breaking changes to the API without creating a new version of the API. Any breaking changes would only be released in a new version of the REST API.

    Non-Breaking Changes
    We consider the following to be examples of non-breaking changes:

    • Adding a new resource
    • Adding a new property to an existing object
    • Adding a new operation (i.e. HTTP POST, PUT, or DELETE) to an existing resource
    • Adding a new object, as long as the object is only used by new resources or new resource operations
    • Adding an alternate set of parameters for an existing resource (overloading)
    • Adding new optional parameters to a resource, where exclusion of those parameters will maintain the same behavior that was present prior to adding the new parameters

    Breaking Changes
    We consider the following to be examples of breaking changes:

    • Addition of a new required input parameter
    • Removal of an existing input parameter (whether required or optional)
    • Removal of a response property
    • Changing the meaning of an existing response property
    • Changing the name or data type of an existing input parameter or output value
    • Removal of an endpoint
    • Changing the response error codes of an existing endpoint
